## S-090: LLM Smart Fill — Natural Language Field Mapping

**Title:** LLM Smart Fill — Generate Field Mappings from Natural Language Description

**Description:**

Add an LLM-powered "describe what you want" flow to accelerate field mapping. User clicks "Smart Fill" button, types a natural language description of desired output (e.g., "extract customer name, email, and last 5 order IDs as a list"), and the system uses Claude API to generate JSONPath selectors and field mappings. User reviews in the visual mapper UI before accepting.

This feature removes friction from the most time-consuming part of the server builder: manual JSONPath extraction. It dramatically speeds up onboarding for non-technical users and reduces the barrier to entry.

**Acceptance Criteria:**

1. A "LLM Smart Fill" button appears in the field mapper UI (near the manual JSONPath entry)
2. Clicking the button opens a modal: "Describe the output you want" with a text area
3. User can type natural language descriptions: "Give me the customer email, order status, and created date"
4. Optional: user can paste a snippet of the actual API response to help the LLM understand structure
5. Clicking "Generate" sends a request to Claude API (claude-3-5-sonnet model or later) with the description and response sample
6. LLM response includes: proposed JSONPath for each field, field name, field description
7. System displays LLM-generated suggestions in the visual mapper with a "Generated by Claude" tag and an "Accept all" button
8. User can: accept individual suggestions, edit them, or discard and return to manual entry
9. Accepted suggestions are added to the field mapping just like manually-entered JSONPaths
10. If LLM API fails (rate limit, quota exceeded, timeout), graceful fallback: "LLM service unavailable. Continue with manual mapping" toast
11. Feature is behind a feature flag (`enable_llm_smart_fill`)
12. LLM API calls are logged (for cost tracking and usage analysis) but do not store user API responses

**Technical Notes:**

- Call Anthropic Claude API (not OpenAI) via `@anthropic-ai/sdk`
- Send: user description + sample JSON response (up to 50KB) + field mapping schema
- Temperature: 0.7 (deterministic but creative)
- Max tokens: 2000
- Rate limit: 100 smart fill requests per user per day (after MVP, per tier)
- Timeout: 15 seconds; if exceeded, return to manual mapping
- Cost: ~$0.002 per call (3.5-sonnet pricing); budget ~$500/month for 250K smart fill calls at 1K MAU scale
- Local LLM option (Ollama) for self-hosted deployments in the future

**Dependencies:**

- Epic 04 field mapping UI must be complete
- Anthropic API key provisioning in platform secrets
- Feature flag system (Unleash or LaunchDarkly)

**Priority:** P1

**Effort:** 8 story points (includes LLM integration, error handling, feature flagging, cost monitoring)

## S-091: XML Response Conversion — Auto-Parse XML to JSON

**Title:** XML Response Conversion — Automatically Convert XML Responses to JSON

**Description:**

Many legacy REST APIs return XML instead of JSON. Support XML by automatically detecting `Content-Type: application/xml` responses, parsing XML using the `quick-xml` crate (Rust), and converting to a JSON-like structure that flows through the document renderer and field mapper.

This unlocks MCP server creation for a large segment of legacy enterprise APIs (SOAP-wrapped REST, older banking APIs, etc.) and significantly expands the addressable market.

**Acceptance Criteria:**

1. When a test call returns `Content-Type: application/xml` or `application/soap+xml`, system detects it
2. XML is parsed using `quick-xml` crate (must handle: CDATA, XML attributes, namespaces, nested elements)
3. Conversion produces a JSON-like structure: `{"root": {"element": [{"name": "value", "@attr": "value"}]}}`
4. Attributes are prefixed with `@` (standard XML-to-JSON convention)
5. CDATA sections are converted to string values (e.g., `<![CDATA[content]]>` → `"content"`)
6. Namespaces are preserved in element names (e.g., `<soap:Body>` → `{"soap:Body": {...}}`)
7. After conversion, response is displayed in the document renderer (same as JSON)
8. Document renderer shows a blue banner: "XML converted to JSON. Original XML is available in the raw response tab."
9. Field mapper works on the converted JSON (user selects fields normally)
10. If XML parsing fails, error message: "Could not parse XML response. Check the raw response tab to debug." + option to use sample JSON fallback
11. Performance: XML parsing must complete within 2 seconds for responses up to 100MB

**Technical Notes:**

- Use `quick-xml` v0.30+ (supports attributes, CDATA, namespaces)
- Conversion logic: build a DOM-like structure, then serialize to JSON
- Store raw XML in the response cache for debugging (visible in "Raw response" tab)
- Test with: AWS XML API responses, SOAP XML, nested namespace scenarios
- No loss of information during conversion (round-trip test: XML → JSON → XML should be lossless)

**Dependencies:**

- Epic 03 test call and document renderer infrastructure
- `quick-xml` Rust crate in gateway dependencies

**Priority:** P1

**Effort:** 5 story points (includes XML parsing, conversion logic, error handling, renderer integration)

## S-092: Docker Export — Generate Standalone MCP Server Project

**Title:** Docker Export — Export MCP Server as Standalone Docker Image

**Description:**

Allow users to export their MCP server configuration as a self-contained Python project (using FastMCP framework) with a Dockerfile, requirements.txt, and README. Users download a ZIP, unzip locally, and run `docker build` and `docker run` to deploy independently of the platform.

This enables: self-hosted deployments for air-gapped environments, local testing, multi-cloud distribution, and reduces platform lock-in concerns for enterprise customers.

**Acceptance Criteria:**

1. On the server detail page, a "Export to Docker" button appears in the actions menu
2. Clicking opens a modal: "Export as Docker project"
3. User can optionally customize: image name (default: `mcp-{server-name}:latest`), Python version (3.9, 3.10, 3.11, 3.12; default 3.11)
4. Clicking "Download" generates a ZIP file containing:
   - `main.py` — FastMCP server code (generated Python, readable and editable)
   - `mcp_server.py` — Tool definitions (auto-generated from config)
   - `auth.py` — Credential and auth configuration (secrets injected via env vars)
   - `Dockerfile` — Multi-stage build (minimal final image, <500MB)
   - `requirements.txt` — Python dependencies (FastMCP, httpx, etc.)
   - `README.md` — Quick start instructions: "docker build -t mcp-server . && docker run -e API_KEY=xxx mcp-server"
   - `.env.example` — Template for environment variables (lists all required API keys/tokens, marked with placeholders)
   - `.gitignore` — Excludes `.env` and Python cache
5. Generated code is production-ready, human-readable (not minified or obfuscated)
6. User can edit the exported code: modify tool names, descriptions, field mappings, error handling
7. Dockerfile uses FastMCP's standard MCP transport (stdio or HTTP) — default stdio for simplicity
8. Credentials are injected via environment variables (e.g., `$API_KEY`, `$BEARER_TOKEN`) — never baked into the image
9. README includes example deployment scenarios: Docker Compose, Kubernetes, AWS ECS
10. File size: ZIP must be <5MB (including all dependencies list, excluding venv)
11. Export works for multi-tool servers; each tool is included in the exported project

**Technical Notes:**

- Use FastMCP Python library (not Rust) for exported servers (lower barrier to modification than Rust)
- Code generation: use Jinja2 templates for Python code scaffolding
- Dockerfile: use `python:3.11-slim` base image (~150MB); install dependencies; copy code; expose MCP transport
- Env var injection: use `os.getenv("API_KEY", None)` with validation and clear error messages
- Testing: manually verify generated code runs locally (test loop: export → unzip → docker build → docker run → test call)
- Support hot reload: mounted config file in Docker (e.g., `-v ./config.json:/app/config.json`) for dev workflows

**Dependencies:**

- Epic 04 server configuration infrastructure
- FastMCP Python SDK installed in templates
- Jinja2 template engine
- Code generation framework

**Priority:** P1

**Effort:** 8 story points (includes code generation, Dockerfile scaffolding, testing, documentation)

## S-095: OAuth 2.0 Support — Authorization Code and Client Credentials Flows

**Title:** OAuth 2.0 Support — Authorization Code and Client Credentials Flows

**Description:**

Upgrade from static credentials (Bearer, API Key, Basic) to support OAuth 2.0 flows, enabling integration with platforms that require token refresh and expiry management (Salesforce, HubSpot, Microsoft Graph, Google APIs, etc.).

Support two common flows: Authorization Code (with PKCE for security) and Client Credentials. Automatically manage token refresh, handle expiry, and securely store refresh tokens.

**Acceptance Criteria:**

1. Auth type selector includes "OAuth 2.0" option (with detailed explanation, not greyed out)
2. When OAuth 2.0 is selected, a radio group appears: "Authorization Code (with PKCE)" or "Client Credentials"
3. **Authorization Code flow:**
   - User enters: client ID, client secret, authorization URL, token URL, scopes (comma-separated)
   - System generates a callback URL (e.g., `https://platform.example.com/oauth/callback`)
   - User clicks "Authorize" → browser redirects to the authorization URL with PKCE challenge
   - User authorizes in the upstream service; browser redirects back with authorization code
   - System exchanges code for tokens (access token + refresh token)
   - Tokens are encrypted and stored
4. **Client Credentials flow:**
   - User enters: client ID, client secret, token URL, scopes (optional)
   - System automatically requests tokens (no user authorization step)
   - Tokens are stored and refreshed automatically
5. Token management:
   - Access tokens are used for API calls
   - Refresh tokens are stored securely (encrypted at rest)
   - System checks token expiry before each API call; if expired, silently refreshes using refresh token
   - If refresh fails, error is displayed: "Re-authorize required. Your session may have expired."
6. Token expiry tracking: system logs expiry time and warns user 1 week before expiration
7. UI displays: "Token valid until [date]" with a "Refresh now" button for manual refresh
8. Scopes are validated against the upstream API's scope list (where available)
9. Testing: OAuth flow can be tested in the server builder before deployment
10. Edge case: if refresh token is revoked by the upstream service, user must re-authorize

**Technical Notes:**

- Use `oauth2` crate (Rust) for OAuth client library
- PKCE: use SHA-256 challenge + verify; generate 128-character random code verifier
- Store: access token (plaintext in memory during request), refresh token (encrypted at rest), token type, expiry time, scopes
- Refresh logic: check expiry 5 minutes before API call; if expired, refresh; if refresh fails, return 401 to client (caller must re-authorize)
- Testing: use local OAuth server (e.g., test Salesforce sandbox) or mock OAuth provider
- Security: ensure redirect URI validation to prevent open redirect attacks

**Dependencies:**

- Epic 02 auth config infrastructure
- OAuth 2.0 libraries (oauth2 crate)
- Token storage and encryption

**Priority:** P1

**Effort:** 8 story points (includes OAuth flows, token management, refresh logic, error handling, security testing)

## S-096: Team Workspaces — Multi-User Collaboration with RBAC

**Title:** Team Workspaces — Shared Workspaces with Role-Based Access Control

**Description:**

Enable teams to collaborate on MCP servers. Create workspaces that can be shared with team members with role-based access: admin (full control), editor (create/edit servers), viewer (read-only access).

This unlocks B2B sales to teams and agencies (not just individual developers) and is essential for enterprise adoption.

**Acceptance Criteria:**

1. User profile page includes "Workspaces" tab
2. By default, each user has one personal workspace (named "[user] workspace")
3. User can create additional workspaces via "+ Create workspace" button
4. Each workspace has: name, description, creation date, member count
5. On workspace detail page, tabs: Overview, Members, Servers, Audit Log, Settings
6. **Members tab:**
   - List of all members with: name, email, role, date joined, action buttons (Edit role, Remove)
   - Three roles:
     - **Admin:** Full control (invite members, change roles, delete workspace, manage billing, delete servers)
     - **Editor:** Can create/edit/delete servers; cannot manage members or workspace settings
     - **Viewer:** Read-only access (view servers, test in playground, but cannot edit)
   - User can invite members by email: "Invite members" input field
   - Invited members receive email with link to join workspace
   - Invitations expire after 7 days if not accepted
7. **Servers tab:** All servers in this workspace are listed here (same view as dashboard, but workspace-specific)
8. **Audit Log tab:** Shows all actions in the workspace: who created/edited/deleted servers, who was added/removed, etc.
9. Credentials are workspace-scoped: if two servers use the same upstream API with the same credentials, they can share one credential entry
10. Cross-workspace access is not allowed: users cannot see servers from other workspaces they're not members of
11. Billing: workspace has one owner (the creator) who is billed for all usage in that workspace
12. Workspace switching: users with multiple workspaces can switch via a workspace selector in the top navigation

**Technical Notes:**

- Add `workspace` table: id, name, description, owner_id, created_at
- Add `workspace_members` table: workspace_id, user_id, role, joined_at
- Server config includes `workspace_id` foreign key
- Credential storage: credentials belong to a workspace (not user), reducing duplication
- Invitation system: temporary tokens (JWT or UUID) sent via email, expire after 7 days
- Audit log: store all mutations (create, update, delete) with user ID, timestamp, object type, before/after state

**Dependencies:**

- Epic 08 server management infrastructure
- User authentication system (assumed to exist)
- Credentials encryption (Epic 09)

**Priority:** P1

**Effort:** 9 story points (includes workspace CRUD, RBAC, invitation system, audit logging, UI for switching)

## S-097: Webhook Notifications — Alert on Errors, Limits, and Expiry

**Title:** Webhook Notifications — Alert on Errors, Rate Limits, and Credential Expiry

**Description:**

Notify users when critical events occur: server errors exceed threshold, upstream API returns consistent errors, credential expiry approaching, rate limit exceeded. Delivery via email (default) or Slack/Discord webhooks (for tech-savvy users).

Keeps teams aware of production issues without constant manual monitoring.

**Acceptance Criteria:**

1. On server detail page, Settings tab includes "Notifications" section
2. Notification types (checkboxes):
   - [ ] Error threshold exceeded (more than 5 errors in 5 minutes)
   - [ ] Consistent API errors (same error type 10+ times in 1 hour)
   - [ ] Credential expiry warning (7 days, 3 days, 1 day before expiry)
   - [ ] Rate limit exceeded (too many calls in short time)
3. Notification channels (radio):
   - **Email** (default): send to workspace owner's email
   - **Slack webhook:** paste Slack webhook URL; system sends formatted Slack message
   - **Discord webhook:** paste Discord webhook URL; system sends formatted Discord message
4. Multiple channels can be enabled simultaneously
5. Webhook URL validation: system validates by sending a test message
6. Notification format examples:
   - **Email:** Subject: "MCP Server Alert: customer-db errors"; Body: "Your customer-db server returned 7 errors in the last 5 minutes. [View details]"
   - **Slack:** Formatted message with server name, error count, timestamp, link to detail page
   - **Discord:** Embed with color-coded severity (red for errors, yellow for warnings)
7. Notification cooldown: no duplicate alerts for the same event within 15 minutes
8. Users can test notifications: "Send test message" button sends a sample alert
9. Notification history: view all notifications sent in the last 30 days (stored in audit log)
10. Opt-out: users can disable notifications for a server globally (checkbox: "Disable all notifications for this server")

**Technical Notes:**

- Event log: each server maintains an event stream (errors, rate limits, etc.)
- Threshold detection: use time-window aggregation (5-minute, 1-hour windows) for error counting
- Webhook delivery: use a background job queue (e.g., Bull with Redis) for reliable webhook delivery (retry up to 3 times with exponential backoff)
- Slack/Discord API: use `incoming-webhooks` (no authentication needed, just POST to webhook URL)
- Credential expiry: run a scheduled job that checks all servers daily and triggers notifications
- Performance: notifications are sent asynchronously (do not block API calls)

**Dependencies:**

- Epic 04 error tracking and logging
- Event stream / message queue system
- Slack/Discord webhook integration

**Priority:** P1

**Effort:** 7 story points (includes event detection, notification dispatch, webhook integration, cooldown logic)

## S-098: Custom Domain Support — Users Point Their Own Domain to MCP Servers

**Title:** Custom Domain Support — Use Your Own Domain for MCP Servers

**Description:**

Allow users to point their own domain to their MCP servers (e.g., `mcp.mycompany.com` instead of `mcp.platform.com/srv/{server-id}`). Includes DNS verification (CNAME), automatic TLS provisioning via Let's Encrypt + Cloudflare, and custom domain management.

Enterprise customers often require white-labeling and domain ownership for security/compliance reasons.

**Acceptance Criteria:**

1. Custom domain support is available on Team and Enterprise tiers only (not Free/Starter)
2. On workspace Settings tab, a "Custom domains" section appears
3. User can add a custom domain: "Add domain" input field for fully-qualified domain name (e.g., `mcp.mycompany.com`)
4. Adding a domain shows DNS setup instructions:
   - "Create a CNAME record pointing your domain to: `platform.example.com`"
   - Copy-to-clipboard button for the CNAME target
5. System periodically checks for DNS propagation (every 30 seconds for first 10 minutes, then hourly)
6. Once CNAME is detected, system automatically provisions a TLS certificate via Let's Encrypt (certificate covers the custom domain)
7. Cloudflare integration (optional): if user provides Cloudflare API token, system can create the CNAME record automatically
8. Certificate auto-renewal: system renews certificates 30 days before expiry (automated via Let's Encrypt ACME)
9. After domain is verified and certificate is active, all API calls to `mcp.mycompany.com` route to the user's servers
10. Subdomain routing: if user has multiple servers, they can use subdomains: `api1.mcp.mycompany.com`, `api2.mcp.mycompany.com`, etc.
11. Domain management: users can view all custom domains, their certificate status, expiry date, and remove domains
12. SSL certificate status is displayed: "Valid until [date]" with green checkmark or "Expired" with red X

**Technical Notes:**

- Use `certbot` or `acme.rs` (Rust library) for Let's Encrypt integration
- DNS verification: use DNS library to query CNAME records (use `hickory-resolver` Rust crate)
- Custom domain routing: configure reverse proxy (Caddy or Nginx) to route traffic based on hostname
- Cloudflare API: use `cloudflare` crate for optional auto-CNAME creation
- Certificate storage: store PEM cert + private key encrypted at rest; reload certificates on renewal
- Security: validate domain ownership (CNAME check) before provisioning certificate

**Dependencies:**

- Reverse proxy infrastructure (Caddy/Nginx)
- Let's Encrypt / ACME protocol support
- DNS verification capability
- Billing tier enforcement

**Priority:** P1

**Effort:** 8 story points (includes CNAME verification, certificate provisioning, auto-renewal, DNS checking)

## S-099: Server Versioning and Rollback — Track and Restore Configs

**Title:** Server Versioning and Rollback — Track Config Changes and Rollback

**Description:**

Track all server configuration changes as versions. Users can view version history with diffs, compare versions, and one-click rollback to any previous version. Auto-saves on every config change prevent accidental data loss.

Critical for production reliability: mistakes can be quickly undone without re-configuring from scratch.

**Acceptance Criteria:**

1. In server detail page, Settings tab includes "Version history" section
2. Every change to server config (URL, auth, fields, tool name, description) creates a new version automatically
3. Version history shows a timeline: most recent at top
4. Each version entry includes: version number (auto-incrementing), timestamp, user who made the change, brief description of what changed (auto-generated, e.g., "Modified field mapping: added 'customer_email'")
5. Clicking a version shows: full config snapshot, user, timestamp, and "View diff" / "Rollback to this version" buttons
6. "View diff" displays a side-by-side diff of the previous version and this version (shows added/removed/modified fields)
7. "Rollback to this version" opens a confirmation dialog: "Are you sure? This will restore [X] field(s) and overwrite current changes."
8. After confirmation, the server reverts to the selected version's config
9. Rollback creates a new version entry (not a true revert): "Rolled back to version #45" in description
10. Version retention: keep at least the last 100 versions per server (or 90 days, whichever is longer)
11. Auto-save: every change (via form field blur or explicit "Save") creates a version (no "discard changes" UI; always saved)
12. Versions are immutable: once created, versions cannot be edited or deleted (except admin deletion of entire server)

**Technical Notes:**

- Version storage: `server_versions` table with: id, server_id, version_number, config (JSON), created_by, created_at, description
- Diff generation: use `json-patch` library (RFC 6902) to compute diffs
- UI: use a diff viewer library (e.g., `react-diff-viewer` for frontend)
- Performance: lazy-load version details (don't fetch all versions at once)
- Cleanup: implement a job to prune versions older than 90 days (keep at least 100 most recent)

**Dependencies:**

- Epic 04 server configuration infrastructure
- Version storage and diff computation

**Priority:** P1

**Effort:** 6 story points (includes version storage, diff generation, UI, rollback logic)

## S-101: API Key Rotation Scheduler — Automated Credential Management

**Title:** API Key Rotation Scheduler — Scheduled Credential Rotation and Renewal

**Description:**

Provide a credential rotation scheduler: users can configure automated rotation of API keys and tokens based on a schedule (e.g., every 30 days, monthly). Integration with upstream API providers' rotation APIs (where supported) for automatic rotation. Manual rotation workflow: generate new credential → test → switch live → revoke old, all with zero downtime.

Improves security posture by reducing key exposure window. Enterprise security teams require regular credential rotation.

**Acceptance Criteria:**

1. On server detail page, Settings tab includes "Credentials" section
2. Each credential displays: name, type (Bearer, API Key, Basic, OAuth), creation date, last rotated date, next rotation date (if scheduled)
3. "Schedule rotation" button allows user to set a rotation interval (7 days, 30 days, 60 days, 90 days, custom)
4. After setting a rotation schedule, system shows: "Next rotation: [date]"
5. **Manual rotation workflow:**
   - User clicks "Rotate now" → a modal opens
   - User enters the new credential value (new API key, new token, etc.)
   - System runs a test call with the new credential to verify it works
   - If test succeeds, user clicks "Switch to new credential" and the system atomically updates the server config
   - Old credential is marked as "revoked" (kept in history for audit, but not used)
   - Service continues without downtime (active connections use old credential; new connections use new)
6. **Automated rotation (if upstream supports):**
   - For supported APIs (Stripe, GitHub, AWS IAM, etc.), system can call the upstream API to generate a new credential automatically
   - User enables "Auto-rotate" toggle for the credential
   - On the scheduled rotation date, system: calls upstream API to generate new credential, tests it, switches to new credential, revokes old
   - User receives notification of successful rotation
7. Rotation history: view all past rotations (date, old credential ID, new credential ID, status, error message if failed)
8. Failed rotation: if test fails, rotation is aborted; user must manually rotate or fix the issue
9. Rotation notifications: user can receive email or Slack notification before rotation and after

**Technical Notes:**

- Supported upstream APIs for auto-rotation (MVP): Stripe, GitHub, AWS IAM, Twilio
- Rotation history: store in `credential_rotations` table with: credential_id, old_value_hash, new_value_hash, rotated_at, rotated_by, status
- Scheduling: use a cron job or background task queue (Bull) to check scheduled rotations daily
- Atomic switch: use a database transaction to update the server config; if switch fails, roll back
- Testing: before switching, perform a test call; timeout is 30 seconds
- Security: never log the plaintext credential values; hash them for verification

**Dependencies:**

- Epic 09 credentials management
- Background job queue for scheduling
- Integration with upstream API providers' credential APIs

**Priority:** P1

**Effort:** 8 story points (includes rotation scheduling, upstream API integrations, atomic switching, failure handling)
